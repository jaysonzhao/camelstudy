Application Migration with Red Hat Fuse Lab

SOAP-based services are plentiful in many enterprise solutions and are slowly being replaced by RESTful services to simplify their use.

In this lab, you develop a Camel route that migrates a legacy SOAP service to a REST API microservice. A new wizard is available in the latest version of Red Hat® Fuse Tooling to help you make the transition with Apache Camel’s Rest DSL. This lab shows how to use the wizard to transition from older SOAP-based services to more modern REST-based services.

Goals

  • Create a REST API for the customer SOAP service from the previous lab

  • Design an Apache Camel route that exposes a REST API consuming JSON

  • Use Red Hat Fuse Tooling and Red Hat Developer Studio to create the wsdl2rest mapping

  • Deploy and test the solution

1. Import Project into Red Hat Developer Studio

  1. Using the same method you used in previous labs, import the 05_migrate_fuse6 use case into a new workspace.

  2. Examine the source code for the use case and expect to find the legacy-soap-rest-wrapper project.

2. Transition from SOAP to REST Service

The SOAP web service from the previous lab, available at 04_enrich-content-REST-and-WS-microservices/soap-cxfws-service, is used as the back-end SOAP service for this lab. This service uses the CXF and Karaf libraries for Red Hat Fuse 6.3, and is available in the soap-cxfws-service project.

  1. Make sure the SOAP service is deployed to Red Hat OpenShift® Container Platform and the WSDL is accessible at the http://$soap-cxfws-service-route/ws/customerService?wsdl^ URL.

  2. Right-click the legacy-soap-rest-wrapper in the Fuse Integration perspective and select New→Camel Rest DSL from WSDL.

  3. Alternatively, navigate to File→New→Other… and select Red Hat Fuse→Camel Rest DSL from WSDL.

  4. For WSDL File, provide the URL to the soap-cxfws-service WSDL, and make sure that the legacy-soap-rest-wrapper project is selected:

  5. Click Next.

  6. Keep the default values of fields and click Finish:

    • Expect to see the Java™ classes and the rest-springboot-context.xml Spring context XML file:

  7. Inspect the Camel route, and note that the solution is code-complete and ready to run.

3. Test REST Service

  1. From the project’s root folder, run the REST service locally: 

    $ mvn spring-boot:run -Dfabric8.skip

    • The Camel service should be running on port 8080, and can be accessed through the http://localhost:8080/api/jaxrs/account^ URL.

  2. Send a test request to the REST service and verify that the back-end service is invoked correctly: 

    $ curl -k http://localhost:8080/api/jaxrs/account -X PUT  -d '{"company":{"name":"Rotobots","geo":"NA","active":true},"contact":{"firstName":"Bill","lastName":"Smith","streetAddr":"100 N Park Ave.","city":"Phoenix","state":"AZ","zip":"85017","phone":"602-555-1100"}}' -H 'content-type: application/json'

    • Expect to see the following response:

      Sample Output
      {"company":{"active":true,"geo":"NA","name":"Rotobots"},"contact":{"city":"Phoenix","firstName":"Bill","lastName":"Smith","phone":"602-555-1100","state":"AZ","streetAddr":"100 N Park Ave.","zip":"85017"},"id":33,"salesContact":"Bernard Tison"}

4. Deploy to OpenShift Container Platform

  1. From a terminal shell, deploy the application to OpenShift Container Platform: 

    $ mvn fabric8:deploy

  2. Verify that the project deployed successfully.

    • Expect to see that a pod for the legacy-soap-rest-wrapper deployment started.

  3. Note the URL of the endpoint for external traffic.

  4. Run the same SOAP UI test the SOAP service as you used previously, replacing the localhost URL with the remote URL.

    • Expect the response to be the same as in localhost.
